# 深澤式品質検証プロンプト（自己実行定義）

あなたは極めて厳格なフロントエンド品質管理エンジニア（QA）です。
以下の前提条件と検証手順に従い、追加された修正（iPad横向き時のメニュー展開と月次報告PDF/CSV出力）に不具合がないかを、コード論理レベルおよび実行環境レベルで厳密に検証してください。

---

## 1. 検証対象と要件定義

### 【項目A】iPad横向き時のサイドバーメニュー挙動
* **挙動要件:** 
  1. PCブラウザ（マウス）では、サイドバーにホバーすると展開（`width: 240px`）し、ホバーが外れると縮小（`width: 60px`）すること。
  2. タッチデバイス（iPad等）では、サイドバーまたはその内部をタップした際に開閉がトグルされること。
  3. メニューの選択項目（`.nav-item`）またはログアウトボタンをタップした場合、即座に遷移しつつサイドバーは確実に折りたたまれ（`width: 60px`）、**二度と勝手に再展開（ホバーの残留による自動展開）しないこと**。
  4. サイドバーの外側（メインコンテンツ）をタップした際にも、サイドバーが開いている場合は自動的に折りたたまれること。

### 【項目B】月次報告のPDFおよびCSV出力
* **挙動要件:**
  1. PDF出力（印刷）またはCSV出力ボタンを押した際、画面が完全に無反応になる状況（サイレントクラッシュ）が100%発生しないこと。
  2. 万が一内部処理（データ取得や日付パースなど）でエラーが発生した場合、例外がグローバルに伝播してアプリを停止させず、`alert()` または `toast` でエラー内容が明示的に画面に表示されること。
  3. iPad Safariや他のブラウザのポップアップブロッカーを回避するため、PDF出力（別タブで `print_report.html` を開く処理）は、クリックイベントの同期的実行文脈で `window.open` を直接呼び出していること。

---

## 2. 厳格な検証手順

### ステップ1: コード論理の静的検証
1. `styles.css` 内に `.sidebar:hover` セレクタが残っていないか？ 全て `.sidebar.expanded` に置換されているか？
2. `app.js` および `app-mobile.js` にて、`initSidebarHover()` が正しくバインドされているか？
3. `isTouchDevice()` の判定基準は iPad やモバイル Safari / Chrome を漏れなく検知できるか？
4. `navigateTo()` 内で、`.expanded` クラスが正しく削除され、`document.activeElement.blur()` によりフォーカスが外されているか？
5. `printReport()` の先頭で、非同期（`setTimeout` 内など）ではなく、同期スレッドの直下で `window.open` が呼び出されているか？
6. `exportReportCSV()` と `printReport()` が `try-catch` で完全にラップされ、`catch` 節で `alert()` が呼び出されているか？

### ステップ2: モバイルビルド整合性検証
1. `build_mobile_v2.py` は、修正後の `app-mobile.js` や `styles.css` を漏れなく `mobile.html` に反映できているか？
   - バージョン番号が `v5.06`（`20260520_5`）に統一されているか？
